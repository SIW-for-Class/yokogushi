# contribution.md
## Yokogushi への関わり方（Contribution Guide）

本ドキュメントは、Yokogushi プロジェクトに **どのように関わり、どのように貢献するか** を整理したガイドである。  
完成度や技術力よりも、**思考の共有と説明**を重視する。

---

## 1. 基本方針

このプロジェクトは教材であり、OSS的な運用を体験する場である。

- 完璧な実装は求めない  
- 小さな改善を歓迎する  
- 「なぜそうしたか」を言語化する  
- 他者の視点を尊重する  

**動くコードより、説明できるコード**が価値を持つ。

---

## 2. 貢献の種類

### 2.1 Issue（提案・質問・観察）
次のような内容は Issue として投稿する。

- 分からない点・疑問
- 挙動の観察結果
- 改善アイデア
- 仕様（spec.md）への提案
- ドキュメントの不足や分かりにくさ

Issue は「問題報告」だけでなく、  
**考えのメモや問いを共有する場所**として使ってよい。

---

### 2.2 Pull Request（変更）
Pull Request（PR）は、以下のような変更を対象とする。

- 表示や文言の小さな改善
- コメントや README の追記
- 仕様に沿った機能追加・修正
- 観察しやすくするための整理（ログ追加など）

**大きな変更は、先に Issue で相談することを推奨**する。

---

## 3. PRの粒度と方針

### 3.1 推奨される粒度
- 1PR = 1テーマ
- 差分は小さく、説明できる範囲に収める

例：
- status 表示文言の変更
- 一覧の並び順変更
- コメントの追加

---

### 3.2 推奨されない例
- 複数の目的が混ざったPR
- 大量のリファクタリングのみ
- 目的や理由が書かれていない変更

---

## 4. PRコメントに書いてほしいこと

PRには、次の3点を必ず書く。

1. **何を変えたか**
2. **なぜ変えたか**
3. **どこを観察してほしいか**

例：
> 一覧表示で status が分かりにくかったため、表示文言を変更しました。  
> DBの status 列と画面表示の対応関係が観察しやすくなることを意図しています。

---

## 5. レビューについて

### 5.1 レビューする側
- 動作確認よりも **意図の理解**を優先する
- 「なぜそう書いたのか」を尋ねる
- 否定ではなく、問いかけを意識する

例：
- 「この処理をここに置いた理由は？」
- 「別の書き方もありそうだけど、どう思う？」

---

### 5.2 レビューされる側
- 指摘は学習の機会と捉える
- すぐに直さなくてもよい（説明できればOK）
- 分からない点は正直に聞いてよい

---

## 6. variants に関するルール

### 6.1 active
- 授業で扱う実装
- 変更は慎重に、小さく
- 仕様逸脱は Issue で事前相談

### 6.2 extensions
- 発展・比較・実験用
- 自由度は高い
- 各実装ディレクトリに README を置く
    - 狙い
    - 観察ポイント
    - 起動方法
## 7. コーディング上の注意（共通）

- 可読性を優先する
- コメントは「何をしているか」より「なぜそうしているか」
- 魔法のような書き方を避ける
- エラーや失敗を隠さない

---

## 8. 非目標（やらないこと）

- 技術力の優劣を競うこと
- パフォーマンス競争
- フレームワーク機能の網羅
- 本番運用レベルの完成度

---

## 9. 最後に

Yokogushi は、

- 読む  
- 観察する  
- 考える  
- 小さく直す  
- 説明する  

という循環を回すための場である。

**遠慮せず、小さく、言葉を添えて**関わってほしい。
